# 技能公共规范

根据真实使用证据持续改进目标 Skill，但始终把准确性、简洁性和可维护性放在新增规则之前。

## 基础规范

除目标 Skill 明确声明例外外，所有引用本技能的子 Skill 必须遵守：

1. **中文优先**：面向用户的说明、报告、代码注释以及新建或修改的 Skill 规则必须使用中文；代码、命令、标识符、协议字段、既有第三方技术文档以及外部工具明确要求的输入可保留原语言。
2. **职责唯一**：子 Skill 必须声明自己负责的能力范围。任务命中该能力时必须调用该 Skill，禁止绕过它改用通用能力、临时流程或其他不相关 Skill。
3. **持续进化**：子 Skill 完成任务并验证后必须调用 `$skill-common` 复盘；没有新增证据时允许不修改，但必须明确记录结论。
4. **依赖去重**：子 Skill 引用其他 Skill 时，只保留调用条件、执行顺序和边界，不得复制被引用 Skill 的规则、清单或实现流程。
5. **修改调用方**：发现子 Skill 与其引用 Skill 能力重合时，只修改当前子 Skill，保留被引用 Skill 不变；删除、合并或改写当前 Skill 的重复内容。
6. **禁止回跳**：被引用 Skill 作为子流程运行时不得反向调用父 Skill，除非父 Skill 明确设计了无循环的协调入口。

新建个人 Skill 时，必须加入对 `$skill-common` 的基础规范引用、专属职责声明和任务结束后的进化入口。只有用户明确说明或工具契约确实冲突时，才允许声明例外，并写明例外范围和原因。

## 核心原则

- 进化不是记录所有问题，而是提炼能降低未来同类失败概率的最小规则。
- Skill 约束不是越多越好；优先修改、合并、删除和下沉，最后才是新增。
- 只根据可验证事实更新 Skill，禁止根据猜测、偏好或单次偶发现象扩张规则。
- 目标 Skill 的核心职责必须保持单一；跨领域能力应调用其他 Skill，而不是复制规则。
- 禁止为了展示“已进化”而强行修改没有真实缺陷的 Skill。

## 必须读取

更新任何 Skill 前，先读取：

1. 目标 Skill 的完整 `SKILL.md`
2. 目标 Skill 在本次任务中实际加载过的相关 `references/`
3. 本次执行的原始证据：错误输出、编译日志、失败步骤、修复差异或用户反馈
4. [references/evolution-policy.md](references/evolution-policy.md)

## 强制工作流程

### 1. 确定进化对象

明确：

- 目标 Skill 路径和名称
- 本次执行的任务目标
- 实际出现的问题
- 问题是否已经解决
- 哪些证据证明问题存在

没有明确目标 Skill 或没有可验证证据时，只输出复盘结论，不修改文件。

### 2. 提取候选经验

为每个问题建立候选项：

| 字段 | 内容 |
|---|---|
| 现象 | 具体失败或遗漏 |
| 根因 | 为什么现有 Skill 未能避免 |
| 修复 | 本次如何解决 |
| 泛化范围 | 哪些未来任务也会遇到 |
| 证据等级 | 硬证据或软证据 |
| 建议动作 | 拒绝、修改、合并、新增、下沉、删除 |

禁止直接把错误信息原文或项目专属业务细节写入通用规则。

### 3. 判断是否值得进化

满足以下任一条件才可进入修改阶段：

- 可稳定复现的编译、测试、校验或运行错误，且根因具有通用性
- 同类软问题在至少两次独立任务中出现
- 现有规则存在明确歧义、冲突、错误或缺失
- 用户明确要求将某项通用约束固化到 Skill
- 已有内容明显重复、过时或阻碍正确执行，需要压缩或删除

以下情况必须拒绝写入：

- 单次业务特例
- 一次性笔误
- 尚未验证的猜测
- 只适用于某个文件名、接口值或临时测试数据的规则
- 已被现有规则完整覆盖的问题
- 仅增加篇幅、不改变决策或执行结果的解释

### 4. 选择最小修改

按以下优先级处理候选经验：

1. **不修改**：现有规则已覆盖
2. **改写**：让已有规则更准确
3. **合并**：并入相邻规则，消除重复
4. **删除**：移除错误、过时或无决策价值的内容
5. **下沉**：细节移入 `references/`
6. **新增**：只有前五种方式都不适用时才增加规则
7. **拆分 Skill**：内容已超出单一职责时，建议创建原子 Skill 并由目标 Skill 引用

禁止将完整的其他 Skill 复制进目标 Skill。应使用 `$skill-name` 说明调用条件和顺序。

修改目标 Skill 前，扫描其全部 `$skill-name` 引用并对比职责。若当前 Skill 重复描述了被引用 Skill 的能力，优先删除或压缩当前 Skill 的重复内容，不得为去重而修改被引用 Skill。

### 5. 执行体积审计

修改前后都要统计目标 Skill：

```bash
wc -l <skill>/SKILL.md
wc -w <skill>/SKILL.md
```

必须遵守：

- 单次进化默认最多新增 3 条独立规则
- 新增内容应尽可能短，每条规则只解决一个决策问题
- `SKILL.md` 超过 300 行时，新增前必须先压缩或下沉
- `SKILL.md` 达到 500 行时，禁止继续追加正文
- 修改后总行数明显增加时，必须说明新增内容为何无法通过改写或合并实现
- 详细示例、长清单和框架差异优先放入一层 `references/`
- 禁止创建 CHANGELOG、复盘日志或按日期无限追加的历史区块

### 6. 保护不可自动修改内容

以下变更必须先征得用户确认：

- 删除或降低安全、隐私、数据保护约束
- 删除强制编译、测试、校验步骤
- 改变目标 Skill 的核心职责或触发范围
- 修改公共工具、凭据或生产环境相关流程
- 将原子 Skill 合并成职责宽泛的大型 Skill
- 一次修改超过 3 条独立规则

可以直接执行的修改：

- 修正文案错误、歧义和失效路径
- 合并重复规则
- 将详细内容下沉到 reference
- 增加有硬证据支撑的小型通用规则
- 修复 `agents/openai.yaml` 与 `SKILL.md` 不一致

### 7. 编辑目标 Skill

编辑时：

- 使用祈使式、可执行的短句
- 说明触发条件、动作和验证方式
- 把规则放到最相关的现有章节
- 保持 frontmatter 的 `name` 不变，除非用户明确要求改名
- 更新 description 时同时检查触发范围是否被意外扩大
- 更新 `agents/openai.yaml`，确保展示名称、简介和默认提示仍与 Skill 一致
- 尊重目标 Skill 现有结构和用户修改，禁止重写无关内容

### 8. 校验与代表性验证

修改后必须：

1. 运行 Skill 官方校验器；缺少依赖时执行等价 YAML 和结构校验
2. 搜索 TODO、旧名称、失效引用和重复规则
3. 检查所有 reference 链接存在且最多一层
4. 对新增规则执行一个最小代表性验证
5. 确认修改没有引入循环 Skill 依赖
6. 再次统计行数，确认体积边界

如果无法进行代表性验证，必须在最终报告中明确说明。

## 与业务 Skill 的组合

业务 Skill 可在任务末尾调用本 Skill：

```text
业务 Skill 执行与验证
    ↓
$skill-common 读取原始证据
    ↓
判断是否具有通用性
    ↓
最小化更新目标 Skill
    ↓
校验与代表性验证
```

不得让两个 Skill 互相要求对方执行，避免循环依赖。推荐只有业务 Skill 单向调用 `$skill-common`。

## 最终报告

必须报告：

- 目标 Skill
- 本次观察到的问题与证据
- 接受和拒绝的候选经验
- 实际采用的动作：改写、合并、新增、下沉或删除
- 修改前后行数
- 是否触及审批边界
- 校验与代表性验证结果
- 未修改时说明拒绝进化的原因
