# 技能进化判定与体积策略

## 证据等级

### 硬证据

出现一次且根因具有通用性时，可以支持进化：

- 编译错误
- 自动化测试失败
- 官方校验器失败
- 可稳定复现的运行时错误
- 明确的 API、格式或工具契约不兼容
- 用户明确指出规则与预期相反，并能由代码或文档验证

硬证据不等于必须新增规则。应先检查现有规则是否已经覆盖，以及失败是否来自未遵守规则。

### 软证据

至少需要两次独立任务出现相同模式：

- 容易遗漏某个步骤
- 输出不够清晰
- 命名或结构不统一
- 执行效率较低
- 某种写法经常导致返工

同一任务中的多次重试只算一次证据。

### 无效证据

- 主观偏好
- 未执行的推测
- 项目专属临时约定
- 无法定位根因的失败
- 已被其他强制规则完整覆盖的问题

## 候选动作决策

| 情况 | 动作 |
|---|---|
| 现有规则已准确覆盖 | 不修改，报告执行偏差 |
| 规则方向正确但有歧义 | 改写 |
| 多条规则表达同一决策 | 合并 |
| 内容错误、过时或无作用 | 删除 |
| 正文细节过多 | 下沉到 reference |
| 通用缺口且无法合并 | 新增 |
| 超出目标 Skill 单一职责 | 拆分新 Skill 并引用 |

## 规则质量检查

每条新增或改写规则必须回答：

1. 在什么条件下触发？
2. 必须执行什么动作？
3. 如何判断执行完成？
4. 它能防止哪类重复失败？
5. 是否与已有规则重复或冲突？

无法回答以上问题的内容不应写入 Skill。

## 体积预算

### 正文预算

- 推荐：`SKILL.md` 不超过 250 行
- 警戒：超过 300 行，必须先压缩再新增
- 上限：500 行，达到上限后禁止继续追加

### 单次进化预算

- 默认最多处理 3 条独立规则
- 默认净增不超过 30 行正文
- 超过预算时，先合并、删除或下沉
- 仍需超过预算时，必须征得用户确认

### Reference 边界

- reference 只保存按需加载的细节、示例、框架差异或长清单
- `SKILL.md` 必须直接链接 reference，禁止多层引用
- reference 不能成为逃避整理的堆积区
- 单个 reference 超过 200 行时，应先去重或按主题拆分

## 规则淘汰

满足以下条件之一时，可建议删除：

- 依赖的工具、API 或项目约定已不存在
- 被更通用且更准确的规则完全替代
- 连续多次执行都不影响任何决策
- 与其他规则冲突且证据表明另一规则正确
- 只是历史说明、事故记录或实现细节

删除安全规则、验证步骤或核心职责约束前，必须获得用户确认。

## 成熟度判断

Skill 趋于成熟的标志：

- 新任务主要复用已有规则
- 新问题更多属于业务特例而非流程缺口
- 规则之间无重复和冲突
- 正文体积稳定
- 代表性任务可一次通过

成熟后应降低进化频率。没有满足证据门槛时，“本次不修改”是正确结果。
